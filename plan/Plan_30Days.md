## 💻 第二阶段：核心项目攻坚 (Day 4 - 20)
**项目**：基于跳表 (SkipList) 的高性能键值存储引擎
**战略**：这是简历上的“核武器”，必须亲手写，不能只看。

### Day 4 - 7: 核心数据结构 (Memory Engine)
- [ ] **Day 4**: 理解跳表 (SkipList) 原理（O(logN) 读写）。
- [ ] **Day 5**: 使用 C++ 实现跳表的 `Insert` 和 `Search`。
- [ ] **Day 6**: 实现 `Delete` 功能，并处理内存释放。
- [ ] **Day 7**: 引入 C++ 智能指针 (`std::shared_ptr`/`unique_ptr`) 优化内存管理。

### Day 8 - 10: 持久化存储 (Persistence)
- [ ] **Day 8**: 设计文件存储格式（简单的 Key-Value 序列化）。
- [ ] **Day 9**: 实现 `load_file`（启动时加载）和 `flush`（写入磁盘）。
- [ ] **Day 10**: 理解文件 IO 操作 (`fstream` vs `open/write`)，尝试优化写入性能。

### Day 11 - 15: 网络层 (High Performance Network) —— **最难最关键**
- [ ] **Day 11**: 学习 Linux Socket 编程基础 (`socket`, `bind`, `listen`, `accept`)。
- [ ] **Day 12**: 彻底理解 **IO 多路复用** (Select vs Poll vs **Epoll**)。
- [ ] **Day 13**: 实现一个基于 **Epoll** 的简单 Echo Server。
- [ ] **Day 14**: 将 Epoll Server 对接到你的 KV 存储引擎上。
- [ ] **Day 15**: 调试网络 Bug，确保客户端可以通过 TCP 发送 "GET key" 并收到回复。

### Day 16 - 20: 并发与压力测试 (Concurrency)
- [ ] **Day 16**: 引入**线程池 (Thread Pool)**，避免为每个请求创建一个线程。
- [ ] **Day 17**: 解决并发安全问题，给跳表加锁（读写锁 `std::shared_mutex`）。
- [ ] **Day 18**: 写一个压测脚本（Client），模拟 1000+ 并发连接。
- [ ] **Day 19**: 观察 QPS (Queries Per Second)，用 GDB 或 perf 分析性能瓶颈。
- [ ] **Day 20**: 整理代码，写好 `README.md`（项目介绍、怎么跑、性能数据）。

## 💡 队长锦囊

1. **关于 AI**：把它当成你的私人导师。遇到报错直接把 Log 甩给它；看不懂的代码直接让它解释。但**绝对不要让它替你写核心逻辑**，否则面试一问就露馅。
2. **关于心态**：前 7 天最痛苦，从 Windows 转 Linux 会有强烈的排斥感。**熬过去，你就赢了一半的人。**
3. **关于完美**：项目不需要完美，不需要像 Redis 一样强大。只要能跑通核心流程（Set/Get），代码逻辑自洽，就是好项目。
